▶ TL;DR
This compute stack is a teaching analogy for a physical supply chain. Scope 1 is the electricity consumed by each forecast request — you own the server, you triggered the compute, just like fuel burned by your own fleet. We measure this live with CodeCarbon on every request. Scope 2 is the grid energy keeping the server alive regardless of traffic — idle draw and infrastructure overhead, like a warehouse that stays lit whether trucks are loading or not. We can calculate this too: a t2.micro running 24/7 in Ohio draws ~3W, costing ~10 kg CO₂eq/year regardless of how many forecasts you run. Scope 3 is everything upstream and downstream: hardware manufacturing, model training, student devices, network transmission, and the AI that built this tool. We measured Scope 1 and estimated Scope 2. Scope 3 is mostly invisible — just like in your suppliers' reports.

▶ How are these emissions calculated?

What we measure

Each forecast request consumes CPU, GPU, and RAM power on the server. CodeCarbon wraps each call, sampling hardware power draw every 1 second, then converts energy consumed into CO2 equivalent:
  • CPU power: Read via Intel RAPL when available; falls back to TDP estimation on cloud VMs.
  • GPU power: Read via NVIDIA NVML when present; zero on CPU-only instances.
  • RAM power: Estimated from total system RAM (~0.3725 W/GB).
  • Conversion: CO2eq = energy (kWh) × grid intensity (kg CO2/kWh)
  • Grid intensity: Auto-detected by region. Ohio: ~0.39 kg CO2/kWh (annual average).

At-scale projections

annual = emissions_per_request × users × 5 requests/user/day × 365
The "5 requests/user/day" is an assumed usage rate for analysts running forecasts. A single request costs almost nothing — the projections show how those tiny costs compound at organizational scale.

Caveats

  • Measurement floor: CodeCarbon's 1-second minimum means even a 10ms forecast gets billed for ~1s of full system power. This overstates per-request emissions for fast methods.
  • Hardware mismatch: Measurements here reflect the current server hardware. A different deployment (e.g. cloud VM) would produce different numbers.
  • Grid intensity is static: Real carbon intensity varies by time of day, season, and renewable generation. CodeCarbon uses a regional annual average.

Why this matters

Every computation has a physical cost. ERP systems, route optimizers, demand planners, ML pipelines — they all consume energy. The question is whether you're measuring it.
